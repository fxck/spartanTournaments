# Tournament App — Domain Glossary

## Competitor
A participant in a tournament. Deliberately generic — can be an individual player or a team of multiple people. In the current boccia tournament (4Kids) a Competitor is a 2-person team, but the model is not restricted to that. Do NOT use "Team" as a synonym in code; use "Competitor" consistently.

## Pairing
A scheduled match between two Competitors on a specific court at a specific time. Belongs to a round and a group. A Pairing has a `gamenumber` (sequential display number) and a `groupID` that encodes which phase it belongs to (see GroupID convention below).

## GroupID convention
- `groupID > 0` → Groups phase pairing, belonging to that group number
- `groupID < 0` → Finals phase pairing; the negative value encodes the round:
  - `-1` = Final
  - `-2` = Semifinal
  - `-4` = Quarterfinal
  - `-8` = Octofinal

## GamePoint
The raw score result of a Pairing: `competitor1Points` and `competitor2Points`. One GamePoint per Pairing (upserted). From a GamePoint, MatchPoints are derived (win/loss/draw).

## MatchPoint
Derived from a GamePoint via `calcMatchPoints()`. Not stored in the DB — always computed. Represents win/loss/draw points for ranking purposes.

## TournamentDetails
Configuration record for a tournament: timing, parallel courts, finalist count, and bcrypt-hashed passwords for Admin and Referee roles. There is exactly one row in this table per deployment. Queries always use `.limit(1)` — no `WHERE id = 1` or similar. The app is intentionally single-tournament — one deployment = one tournament.

## Tournament phases
1. **Groups phase** — Round-robin within groups. Groups are formed by assigning Competitors via a random draw (`drawNumber`). Pairings are generated by `CalcMostGamesPerCompetitorPlan`.
2. **Finals phase** — Knockout bracket seeded from group standings. Generated by `calcFinals()`, progressed round-by-round by `calcNextFinalRound()`.

## Roles
- **Admin** — Configures TournamentDetails, manages Competitors, triggers tournament calculation.
- **Referee** — Enters and edits GamePoints for Pairings.
- **Public** — Read-only: views current/upcoming games, group standings, full schedule, individual Competitor view.

Auth flow: Client POSTs plaintext password over HTTPS to a tRPC mutation. Server bcrypt-compares against stored hash, sets a signed session cookie on success. The password never leaves the server after login. Role is encoded in the session cookie. Client never computes hashes. Session managed via `iron-session` (encrypted cookie, no server-side store).

Passwords are stored as bcrypt hashes in `tournamentDetails`. They are set via:
- **Initial setup:** A setup page shown when no `tournamentDetails` row exists yet.
- **Change:** Optional "Neues Passwort" field in the Admin form — empty = keep current, filled = hash and store.
- **Reset/forgotten:** DELETE the `tournament_detail` row → app shows setup page again.

There is no `users` table. The only identities in the system are Admin and Referee, both defined via passwords stored in TournamentDetails.

## Setup guard
If no `tournamentDetails` row exists, all routes redirect server-side to `/setup`. The setup page is only accessible when no tournament exists — after successful setup it redirects to `/admin`.

## Route protection
- Admin pages: Angular `canActivate` Route Guard calls `auth.session` tRPC query → redirects to `/login` if no valid session. tRPC mutations additionally verify session server-side (defense in depth).
- Referee actions (GamePoint entry): UI elements hidden/disabled client-side via session signal; server-side tRPC mutation rejects without referee/admin session.
- Nav shows Admin link only when session has `role: 'admin'`.
- Session managed via `iron-session` (encrypted cookie).

## Pages & Navigation
Persistent top navigation bar with:
- Home — current & upcoming games (public)
- Spielplan — full schedule, groups + finals (public)
- Gruppen — group standings with match points (public)
- Ergebnisse — game results; referees can enter/edit (role-gated UI)
- Competitor/[id] — personal view: tabs for Spielplan / Gruppen / Ergebnisse filtered to that competitor
- Admin — tournament config + competitor management (admin-only, hidden from nav if not logged in)

## AppState
No global AppState service. Data loading uses Analog-native patterns:
- **`injectLoad()`** — route-level server data, runs during SSR, available immediately on render
- **`resource()`** — reactive client-side fetching, used after mutations or for polling (e.g. home page auto-refresh)
- Derived views (groups, filtered pairings, finals) are local computed signals
- The Competitor filter is a route parameter, not shared state
- Auth status (role) comes from the session cookie, read server-side

## API Layer
Nitro Routes in `server/routes/api/`. No tRPC. Type safety via Zod validation in each handler + shared TypeScript types for request/response shapes. Auth enforced per-route via `requireSession()` helper.
